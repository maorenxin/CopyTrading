# 任务: [FEATURE NAME]

**输入**: 设计文档来自 `/specs/[###-feature-name]/`
**前置条件**: plan.md（必需）, spec.md（用户故事必需）, research.md, data-model.md, contracts/

**测试**: 下述示例包含测试任务。测试为可选 - 仅在功能规格明确要求时添加。

**组织方式**: 按用户故事分组，以便每个故事独立实现与测试。

## 格式: `[ID] [P?] [Story] 描述`

- **[P]**: 可并行（不同文件，无依赖）
- **[Story]**: 任务所属用户故事（例如 US1, US2, US3）
- 描述中必须包含具体文件路径

## 路径约定

- **单项目**: `src/`, `tests/` 在仓库根目录
- **Web 应用**: `backend/src/`, `frontend/src/`
- **移动端**: `api/src/`, `ios/src/` 或 `android/src/`
- 下方路径示例以单项目为假设 - 请根据 plan.md 调整

## 依赖与执行顺序

### 阶段依赖

- **设置（Phase 1）**: 无依赖 - 可立即开始
- **基础能力（Phase 2）**: 依赖设置完成 - 阻塞所有用户故事
- **用户故事（Phase 3+）**: 依赖基础能力完成
  - 可并行（团队允许）
  - 或按优先级顺序 (P1 → P2 → P3)
- **打磨（最终阶段）**: 依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1 (P1)**: 基础能力完成后可开始 - 不依赖其他故事
- **用户故事 2 (P2)**: 基础能力完成后可开始 - 可能集成 US1，但应独立测试
- **用户故事 3 (P3)**: 基础能力完成后可开始 - 可能集成 US1/US2，但应独立测试

### 单个用户故事内

- 测试（如包含）必须先写并失败，再实现
- 模型先于服务
- 服务先于接口
- 核心实现先于集成
- 当前故事完成后再进入下一优先级

### 并行机会

- 所有标记 [P] 的设置任务可并行
- Phase 2 中标记 [P] 的基础任务可并行
- 基础能力完成后，多个用户故事可并行
- 单个故事内标记 [P] 的测试可并行
- 单个故事内标记 [P] 的模型可并行
- 不同用户故事可由不同成员并行

## 实施策略

### MVP 优先（仅用户故事 1）

1. 完成 Phase 1: 设置
2. 完成 Phase 2: 基础能力（关键 - 阻塞所有故事）
3. 完成 Phase 3: 用户故事 1
4. **停止并验证**: 独立测试用户故事 1
5. 如可行，部署/演示

### 增量交付

1. 完成设置 + 基础能力 → 基础可用
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事都应独立交付价值，且不破坏之前故事

### 并行团队策略

多开发者时:

1. 团队完成设置 + 基础能力
2. 基础能力完成后:
   - 开发者 A: 用户故事 1
   - 开发者 B: 用户故事 2
   - 开发者 C: 用户故事 3
3. 各故事独立完成并集成

## 备注

- [P] 任务 = 不同文件、无依赖
- [Story] 标签 = 映射到具体用户故事
- 每个用户故事都应独立完成与测试
- 测试需先失败再实现
- 每个任务或合理任务组后提交
- 在检查点停下以独立验证
- 避免: 含糊任务、同文件冲突、跨故事依赖
