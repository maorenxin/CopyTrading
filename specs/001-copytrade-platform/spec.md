# 功能规格: 跟单平台多端交互与跟单流程

**功能分支**: `001-copytrade-platform`  
**创建时间**: 2025-12-24  
**状态**: 草案  
**输入**: 用户描述: "多端自适应的跟单平台，支持卡片/表格视图、交易员详情、跟单确认、多语言切换，以及 Telegram 机器人操作；数据来自 Hyperliquid vault，并收取 0.1% 手续费。"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 发现与对比交易员（优先级: P1）

用户在网页端浏览交易员列表，通过卡片或表格视图对比业绩，并可按 Sharpe、All-Time Return、最大回撤等排序，在移动端也能顺畅查看与切换。

**为何此优先级**: 发现与对比是进入跟单的第一步，决定用户是否继续深入。

**独立测试**: 打开首页即可完成视图切换、排序、进入交易员详情并返回。

**验收场景**:

1. **假设** 用户进入首页，**当** 切换卡片/表格并选择排序指标，**则** 列表以所选指标排序展示。
2. **假设** 用户使用手机访问，**当** 查看“my portfolio”等受限区域，**则** 通过点击进入专用页面并可返回原列表。

---

### 用户故事 2 - 查看详情并发起跟单（优先级: P2）

用户点击交易员进入详情页，查看指标、交易历史与净值对比，并在确认弹窗中选择或输入跟单金额后提交。

**为何此优先级**: 直接产生交易与收入，是核心业务价值。

**独立测试**: 进入任意交易员详情即可完成指标查看与跟单确认流程。

**验收场景**:

1. **假设** 用户进入交易员详情，**当** 切换 7D/30D/90D/All 指标，**则** 对应指标与图表同步更新。
2. **假设** 用户点击“跟单”，**当** 选择默认或自定义金额并确认，**则** 系统记录并展示跟单结果。

---

### 用户故事 3 - Telegram 机器人操作（优先级: P3）

用户通过 Telegram 与机器人对话完成下单、查看收益、撤单与调整杠杆等操作，并获得与网页一致的核心数据。

**为何此优先级**: 提供移动入口与即时操作，是多端体验的关键部分。

**独立测试**: 与机器人对话即可完成一次下单或收益查询。

**验收场景**:

1. **假设** 用户在 Telegram 发起查询，**当** 请求收益或订单状态，**则** 返回与网页一致的核心数据。

---

### 边界情况

- 当移动端屏幕过窄时，列表与详情如何保证可读性与可返回？
- 当某个交易员数据暂不可用时，如何展示并避免误导？
- 当用户未连接钱包或未授权时，跟单按钮如何处理？
- 当语言切换时，是否需要保持当前页面与筛选状态？

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系统必须提供交易员列表的卡片视图与表格视图，并允许一键切换。
- **FR-002**: 卡片视图必须展示交易员地址、五维雷达图、胜率等关键指标。
- **FR-003**: 表格视图必须覆盖与卡片等价的核心指标信息。
- **FR-004**: 系统必须支持按 Sharpe、All-Time Return、最大回撤等指标排序。
- **FR-005**: 交易员详情页必须展示交易历史、7D/30D/90D/All 指标切换、净值走势与比特币持有对比。
- **FR-006**: 移动端必须自适应布局；“my portfolio”等区域在小屏幕上需通过点击进入独立页面并支持返回。
- **FR-007**: 系统必须提供中英文切换入口，并在切换时保持当前页面状态。
- **FR-008**: 跟单确认弹窗必须支持默认 100 USDC、快捷选项 500/1000/max，以及自定义金额输入。
- **FR-009**: 系统必须清晰展示跟单手续费比例为 0.1%，并在用户确认前提示。
- **FR-010**: 系统必须基于 Hyperliquid vault 的交易数据持续更新展示内容。
- **FR-011**: Web 端与 Telegram 端必须保持核心数据、筛选与排序的一致性。
- **FR-012**: 系统必须支持通过 Telegram 机器人完成下单、查看收益、撤单与调整杠杆的核心操作。
- **FR-013**: 系统必须采用用户授权平台代为交易的模式完成下单与资金处理。
- **FR-014**: 系统必须在下单前从跟单金额中扣除 0.1% 手续费，并向用户明确展示扣费结果。
- **FR-015**: Telegram 端必须通过网页深链签名完成身份绑定，以保证交易授权安全。

### 宪章要求 *(必填)*

- 指标必须可复现，并记录公式、数据来源与时间窗口。
- 钱包连接是唯一认证方式，不得出现用户名/密码账户体系。
- 任何可能导致资金移动的操作都需要明确的钱包签名，且不得自动执行。
- Web UI 与 Telegram 可调用 API 必须提供等价的核心数据与筛选。
- 用户可见文案必须包含风险披露并说明非投资建议。

### 关键实体 *(如涉及数据则必须)*

- **交易员**: Hyperliquid vault 地址与其交易表现集合。
- **绩效指标**: 回报率、Sharpe、胜率、回撤、时间窗口等汇总数据。
- **跟单订单**: 用户、金额、目标交易员、状态与时间戳。
- **用户钱包**: 钱包地址与语言偏好等必要信息。
- **交易历史**: 交易员的历史交易记录与净值走势数据。

### 假设与依赖

- 当前版本仅支持中文与英文。
- 货币单位以 USDC 为默认展示单位。
- 交易员与策略均来源于 Hyperliquid vault。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 80% 的新用户能在 3 分钟内完成“找到交易员并进入详情”的流程。
- **SC-002**: 70% 的已连接钱包用户可在 2 分钟内完成一次跟单确认。
- **SC-003**: 移动端用户在 2 分钟内完成列表浏览与详情返回的成功率达到 85%。
- **SC-004**: 90% 的 Telegram 用户可在 2 分钟内完成一次查询或下单指令。
- **SC-005**: 交易员展示数据更新滞后不超过 5 分钟。
