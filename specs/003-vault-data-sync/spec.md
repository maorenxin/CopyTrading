## 用户场景与测试 *(必填)*

### 用户故事 1 - 本地同步 Vault 数据（优先级: P1）

运营人员希望将 Hyperliquid 上公开的 Vault 数据同步到本地数据环境，覆盖
Vault 基本信息、历史成交、当前仓位、存款人等核心数据。

**为何此优先级**: 数据同步是后续分析与产品展示的前提，必须最先完成。

**独立测试**: 触发一次同步后，本地数据库中能查到每个 Vault 的基础信息、
成交记录、仓位与存款人数据，并具备同步时间标记。

**验收场景**:

1. **假设** 本地数据为空，**当** 触发一次同步，**则** 所有公开 Vault 都
   产生完整的基础信息记录并包含同步时间。
2. **假设** Vault 有成交历史，**当** 同步完成，**则** 数据库中存在该 Vault
   的成交明细与时间顺序。

---

### 用户故事 2 - 实时更新与去重写入（优先级: P2）

运营人员希望在同步后持续通过 WebSocket 监听重点 Vault 的交易变化，并将
最新成交实时写入本地数据库，且对重复 Vault 地址执行覆盖更新。

**为何此优先级**: 需要保持数据最新，同时确保重复写入不会污染数据质量。

**独立测试**: 触发同步后，对 Top10 Vault 建立实时监听，产生新成交即刻
写入数据库，并确保重复地址被更新而非重复插入。

**验收场景**:

1. **假设** 某 Vault 已存在记录，**当** 同步再次写入该地址，**则** 原记录
   被覆盖更新且无重复条目。
2. **假设** 已建立 WebSocket 监听，**当** 监听到最新成交，**则** 成交明细
   在数据库中新增并可被查询。

---

### 用户故事 3 - 回退数据源与可追溯性（优先级: P3）

运营人员在官方数据源不可用时，仍希望通过公开页面获得 Vault 数据，保证
平台持续运行，并能追踪每次同步的范围、时间与来源。

**为何此优先级**: 需要可用的回退路径与可追溯性，降低数据中断风险。

**独立测试**: 模拟官方数据源不可用时，同步仍可完成且产出关键字段，并有
清晰的同步来源记录。

**验收场景**:

1. **假设** 官方数据源不可访问，**当** 触发同步，**则** 系统通过公开页面
   获取 Vault 数据并写入本地数据库，且同步日志标注来源与时间。

---

### 边界情况

- 当 Vault 列表为空或部分 Vault 暂无成交时，系统如何记录与标注空数据？
- 当单个 Vault 数据缺失（如仓位或存款人信息）时，系统如何继续同步并记录缺口？
- 当同步中断或超时，系统如何报告已完成的部分与失败原因？
- 当数据源结构发生变化时，系统如何提示字段映射风险？

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系统必须同步所有公开 Vault 的基础信息，并记录同步时间。
- **FR-002**: 系统必须同步 Vault 的历史成交、当前仓位与存款人数据。
- **FR-003**: 系统必须优先使用官方数据源获取 Vault 数据，若不可用则使用
  公开页面作为回退数据源。
- **FR-004**: 系统必须将数据写入与线上 Supabase 结构一致的本地数据环境，
  以便未来迁移。
- **FR-005**: 系统必须支持重复同步且对重复 Vault 地址执行覆盖更新。
- **FR-006**: 系统必须通过 WebSocket 监听指定 Vault 的最新交易并实时写入。
- **FR-007**: 系统必须记录每次同步的时间范围、来源与结果摘要。
- **FR-008**: 系统必须支持部分失败的同步，并能区分成功与失败的 Vault。

### 宪章要求 *(必填)*

- 指标必须可复现，并记录公式、数据来源与时间窗口。
- 钱包连接是唯一认证方式，不得出现用户名/密码账户体系。
- 任何可能导致资金移动的操作都需要明确的钱包签名，且不得自动执行。
- Web UI 与 Telegram 可调用 API 必须提供等价的核心数据与筛选。
- 用户可见文案必须包含风险披露并说明非投资建议。

### 关键实体 *(如涉及数据则必须)*

- **Vault**: 单个 Vault 的基础信息（地址、名称、策略摘要等）。
- **VaultTrade**: Vault 的历史成交明细（时间、方向、价格、数量等）。
- **VaultPosition**: Vault 当前持仓与净值相关字段。
- **VaultDepositor**: Vault 存款人列表与资金占比信息。
- **SyncRun**: 每次同步执行记录（时间、来源、结果、范围）。

### 依赖与假设

- 外部数据源提供公开可访问的 Vault 信息与历史数据。
- 本地数据环境与线上 Supabase 结构保持一致，以支持未来迁移。
- 同步范围仅覆盖公开可见的 Vault，不包含需授权的私有数据。
- 同步操作可由人工触发或按需执行，不要求自动化调度。
- WebSocket 监听范围优先覆盖年化收益排名前 10 的 Vault。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 对于公开 Vault 列表，100% 的 Vault 在本地数据库中至少有一条
  基础信息记录。
- **SC-002**: 对于存在成交历史的 Vault，成交记录同步后可完整覆盖官方数据
  中的最近记录（允许因数据源限制产生明确标注）。
- **SC-003**: 连续两次同步在无新数据时，重复记录占比低于 1%。
- **SC-004**: 在接近 30000 个 Vault 的规模下，一次完整爬取可在 60 分钟内
  完成（允许在同步记录中注明异常）。
- **SC-005**: 年化收益排名前 10 的 Vault 已建立 WebSocket 监听并持续写入
  最新成交。
- **SC-006**: 每次同步都会生成可追溯的结果摘要，包含成功 Vault 数量与失败
  Vault 列表。
