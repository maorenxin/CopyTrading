# 功能规格: Web3 钱包连接与登录

**功能分支**: `002-wallet-connect`  
**创建时间**: 2025-12-24  
**状态**: 草案  
**输入**: 用户描述: "新增 Web3 钱包连接功能：在 Chrome 里可唤起 OKX/MetaMask 等钱包完成连接与登录；连接后显示钱包地址、支持断开；未连接时提示连接。在当前的 WalletConnect.tsx 中继续完善"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 连接钱包（优先级: P1）

用户在网页中点击“连接钱包”，浏览器唤起 OKX/MetaMask 等钱包扩展完成授权，页面展示已连接状态与钱包地址。

**为何此优先级**: 钱包连接是所有后续操作的前提。

**独立测试**: 用户点击按钮完成一次连接即可验证。

**验收场景**:

1. **假设** 用户未连接钱包，**当** 点击连接按钮并完成授权，**则** 页面展示钱包地址与“已连接”状态。
2. **假设** 用户取消授权，**当** 返回页面，**则** 仍显示“未连接”并提示重试。

---

### 用户故事 2 - 断开连接（优先级: P2）

用户在已连接状态下点击“断开连接”，页面恢复未连接状态并隐藏地址。

**为何此优先级**: 用户需要可控的退出与隐私管理。

**独立测试**: 已连接用户点击断开即可验证。

**验收场景**:

1. **假设** 用户已连接钱包，**当** 点击断开，**则** 地址被清除且显示“未连接”。

---

### 用户故事 3 - 连接状态提示（优先级: P3）

用户访问页面时若未连接钱包，系统提示可连接；若钱包不可用则提示安装扩展。

**为何此优先级**: 降低用户进入门槛，减少迷惑。

**独立测试**: 在无钱包环境打开页面可看到提示文案。

**验收场景**:

1. **假设** 浏览器未检测到钱包，**当** 用户进入页面，**则** 显示安装提示与说明。

---

### 边界情况

- 当用户拒绝授权时，页面如何提示并允许重试？
- 当用户切换钱包账户时，页面是否更新地址？
- 当用户在多个标签页打开页面时，连接状态是否保持一致？

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系统必须在 Chrome 中唤起钱包扩展完成连接授权。
- **FR-002**: 系统必须支持 OKX 与 MetaMask 等常见钱包扩展。
- **FR-003**: 连接成功后必须显示钱包地址与已连接状态。
- **FR-004**: 用户必须可以主动断开连接，并恢复未连接状态。
- **FR-005**: 当钱包不可用时，系统必须提示用户安装钱包扩展。
- **FR-006**: 连接失败或用户拒绝授权时，系统必须提示失败原因并允许重试。
- **FR-007**: 钱包地址显示必须进行简化（例如 0x1234...abcd）。
- **FR-008**: 连接状态必须在页面刷新后可恢复（基于本地缓存或钱包状态）。

### 宪章要求 *(必填)*

- 指标必须可复现，并记录公式、数据来源与时间窗口。
- 钱包连接是唯一认证方式，不得出现用户名/密码账户体系。
- 任何可能导致资金移动的操作都需要明确的钱包签名，且不得自动执行。
- Web UI 与 Telegram 可调用 API 必须提供等价的核心数据与筛选。
- 用户可见文案必须包含风险披露并说明非投资建议。

### 关键实体 *(如涉及数据则必须)*

- **钱包会话**: 当前连接的钱包地址与连接状态。

### 假设与依赖

- 浏览器安装了 OKX 或 MetaMask 扩展。
- 当前实现聚焦 Web 端，Telegram 端不包含钱包连接。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 80% 的用户能在 1 分钟内完成一次钱包连接。
- **SC-002**: 断开连接后 100% 的用户看到未连接状态且地址被清除。
- **SC-003**: 钱包不可用时，提示文案展示成功率达到 100%。
- **SC-004**: 连接成功后地址显示正确率达到 99%。
