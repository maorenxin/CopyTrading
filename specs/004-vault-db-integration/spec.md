## 用户场景与测试 *(必填)*

### 用户故事 1 - 数据批次落库与保留 CSV（优先级: P1）

数据运营在本地运行 vault 下载与指标计算流程，期望同一批次的数据既保留 CSV 文件，又写入本地数据库，便于后续核对与复用。

**为何此优先级**: 这是后续联调与数据一致性保障的基础，没有落库就无法替换前端 mock 数据。

**独立测试**: 通过执行一次完整数据流程，验证 CSV 仍按现有方式产出，且数据库存在同批次对应记录。

**验收场景**:

1. **假设** 本地环境可运行数据流程且数据库可用, **当** 完成一次下载与指标计算, **则** 生成与现有规则一致的 CSV 且数据库中存在同一批次的数据记录
2. **假设** 同一批次再次运行, **当** 重复执行数据流程, **则** 前端不会出现重复数据且该批次可被追溯

---

### 用户故事 2 - 前端读取真实数据（优先级: P2）

前端用户访问 vault 相关页面时，看到的应是最新真实数据而不是 mock 数据，且在数据不可用时得到明确提示。

**为何此优先级**: 上线后必须依赖真实数据，保证用户看到的内容与实际一致。

**独立测试**: 切换前端数据源后访问页面，验证数据来自数据库且非 mock，并在无数据时出现空态提示。

**验收场景**:

1. **假设** 数据库中已有最新批次数据, **当** 用户访问 vault 列表与详情页面, **则** 页面显示真实数据且来源与批次可追溯
2. **假设** 数据库暂无可用数据, **当** 用户访问相关页面, **则** 页面展示空态或提示并不显示 mock 数据

---

### 用户故事 3 - CSV 与数据库一致性核对（优先级: P3）

数据核对人员需要依据同一批次的 CSV 与数据库内容进行比对，快速确认是否存在缺失或差异。

**为何此优先级**: 保障数据可信度，避免因落库异常影响前端展示。

**独立测试**: 选取一个批次，核对 CSV 与数据库的记录数和关键字段一致性，输出核对结论。

**验收场景**:

1. **假设** 某批次已产出 CSV 且已入库, **当** 进行一致性核对, **则** 可得到记录数与关键字段差异统计

---

### 边界情况

- 下载过程中部分 vault 失败时，应如何处理该批次的落库与前端展示
- 指标计算出错或缺少必要数据时，是否允许该指标进入数据库与前端
- 重复导入同一批次导致冲突时，系统如何避免重复展示或数据污染
- 数据结构新增字段时，CSV 与数据库如何保持一致可对账

### 范围与不在范围

- 范围包含：vault 下载与指标计算结果的落库、CSV 保留、对账摘要产出、前端由 mock 切换到真实数据
- 不在范围：新增业务指标口径、非 vault 相关页面的新功能、权限与认证机制的改造

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系统必须保持现有 CSV 输出行为，确保每次数据流程都生成并保留 CSV 文件以便对账
- **FR-002**: 系统必须将下载与计算结果写入本地数据库，并与 CSV 的批次信息可一一对应
- **FR-003**: 系统必须记录每条指标的公式/版本、数据来源与时间窗口，以支持结果可复现
- **FR-004**: 系统必须提供前端可调用的数据能力，覆盖现有使用 mock 的 vault 列表、详情与指标展示
- **FR-005**: 前端必须默认使用真实数据源，不依赖 mock 数据；当无可用数据时需展示空态或提示
- **FR-006**: 系统必须提供可用于 CSV 与数据库一致性核对的摘要结果（批次、记录数、差异计数）
- **FR-007**: 当下载或计算失败时，系统不得写入不完整数据，并应保留失败信息以便追溯
- **FR-008**: 重复执行同一批次时，系统必须避免产生重复展示或不可追溯的数据冲突

### 需求验收要点

- **FR-001**: 执行一次完整流程后，CSV 生成且历史 CSV 不被删除
- **FR-002**: 同批次数据可在数据库中检索，并能与 CSV 批次信息对应
- **FR-003**: 随机抽取指标记录可追溯到公式/版本、数据来源与时间窗口
- **FR-004**: 前端能获取 vault 列表、详情与指标，且数据与数据库一致
- **FR-005**: 前端不再依赖 mock，且在无数据时显示空态或提示
- **FR-006**: 任一批次可产出对账摘要，包含记录数与差异计数
- **FR-007**: 人为触发下载或计算失败时，不写入不完整数据且失败可追溯
- **FR-008**: 同批次重复执行后，前端无重复展示且批次可追溯

### 宪章要求 *(必填)*

- 指标必须可复现，并记录公式、数据来源与时间窗口。
- 钱包连接是唯一认证方式，不得出现用户名/密码账户体系。
- 任何可能导致资金移动的操作都需要明确的钱包签名，且不得自动执行。
- Web UI 与 Telegram 可调用 API 必须提供等价的核心数据与筛选。
- 用户可见文案必须包含风险披露并说明非投资建议。

### 依赖与假设

- 本地环境具备可用的数据存储服务并允许写入
- 现有 CSV 输出路径与命名规则稳定且可用于对账
- 现有前端 mock 数据覆盖范围清晰，默认包含 vault 列表、详情与指标页面
- 下载与指标计算流程可按批次重复运行

### 关键实体 *(如涉及数据则必须)*

- **Vault**: 代表可被跟踪的 vault 实体，包含唯一标识、名称、来源平台与基础属性
- **Vault 指标**: 代表某个 vault 在指定时间窗口的量化结果，关联 vault、公式/版本与数据来源
- **数据批次**: 代表一次完整采集与计算的执行单位，包含批次时间、覆盖范围与状态
- **对账摘要**: 代表 CSV 与数据库的对比结果，包含批次、记录数与差异统计

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 每个数据批次完成后，CSV 与数据库记录数一致率达到 100%，关键字段差异率不高于 0.5%
- **SC-002**: 前端核心页面（vault 列表、详情、指标）至少 95% 的访问展示真实数据而非 mock
- **SC-003**: 数据批次完成后 10 分钟内，前端可看到该批次的最新数据
- **SC-004**: 运营人员在 5 分钟内可完成一次“采集-入库-对账”验收流程
